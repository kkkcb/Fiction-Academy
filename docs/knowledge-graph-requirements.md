# 知识图谱融合需求文档

## 一、背景与目标

### 1.1 当前痛点

Fiction Academy 作为小说创作辅助平台，目前的核心能力是：
- **对话式创作**：用户与 AI 对话，逐步构建小说世界
- **工作台**：手动或 AI 辅助整理世界观、角色、大纲
- **待确认机制**：AI 反向提问，帮助用户补充细节

但存在以下问题：

1. **信息碎片化**：用户在多轮对话中散落了大量设定信息（角色关系、地点、事件），缺乏结构化关联
2. **一致性难维护**：角色 A 说住在"西湖边"，后面又提到"住在北京"，没有自动检测矛盾
3. **关系网络不可视化**：角色之间的关系（师徒、敌对、暗恋）、地点之间的地理关系、事件之间的因果链条，都是纯文本，无法直观理解
4. **设定难以复用**：同一个世界观下的多部作品、同一个角色在不同场景下的表现，无法联动
5. **AI 创作缺乏约束**：AI 不知道角色 A 和角色 B 之间的关系，可能写出违背设定的内容

### 1.2 目标

引入**知识图谱**作为小说创作的"结构化大脑"：

- **不是**让用户手动构建图谱（太复杂）
- **而是**让 AI 在对话中自动提取实体和关系，构建和维护知识图谱
- 用户通过**可视化图谱**直观查看和编辑角色关系、世界观架构
- 图谱作为 AI 的**结构化记忆**，约束后续创作的一致性

### 1.3 核心理念（来自文章的方法论）

参考蚂蚁知识图谱方法论中的核心思想：

| 文章方法论 | 本项目映射 |
|-----------|-----------|
| **结构与语义解耦** | Schema（角色/地点/事件的属性定义） vs 概念（修仙体系/势力等级等分类体系） |
| **实体-关系设计** | 角色-关系→角色、角色-所属→势力、事件-发生在→地点 |
| **概念语义建模** | 修仙境界体系（练气→筑基→金丹...）、势力等级（一流→顶级→超级） |
| **属性语义标化** | 角色的"境界"属性自动关联到境界概念树，实现"同境界"角色发现 |
| **多元关系（事件）** | "张三在青云峰击败李四"是一个事件，含主体、客体、地点、时间、行为 |
| **事理关系** | 事件之间的因果、顺承、触发关系（击杀师父→叛出师门→江湖追杀） |

## 二、功能需求

### 2.1 知识图谱数据模型

#### 2.1.1 实体类型（Schema）

```
角色（Character）
├── id, name, alias（别名）
├── gender（性别）
├── age（年龄/年龄段）
├── realm（境界）        → 语义标化：关联境界概念树
├── faction（所属势力）   → 语义标化：关联势力实体
├── occupation（职业）
├── personality（性格描述）
├── appearance（外貌描述）
├── abilities（能力列表）
└── background（背景故事）

地点（Location）
├── id, name
├── type（大陆/国家/城市/山脉/秘境/宗门）
├── parent（上级地点）    → 语义标化：地理层级
├── description
└── features（特征标签）

势力（Faction）
├── id, name
├── level（等级）        → 语义标化：关联势力等级概念树
├── location（总部位置）  → 语义标化：关联地点实体
├── leader（首领）       → 语义标化：关联角色实体
├── members（成员列表）
└── description

物品（Item）
├── id, name
├── type（武器/丹药/法宝/功法/材料）
├── owner（持有者）      → 语义标化：关联角色实体
├── rank（品阶）         → 语义标化：关联品阶概念树
└── description

事件（Event）           ← 多元关系节点
├── id, name
├── type（战斗/交易/修炼/突破/结盟/背叛/死亡...）
├── subject（主体）     → 关联角色
├── object（客体）     → 关联角色/势力/物品
├── location（地点）   → 关联地点
├── time（时间描述）
├── cause（原因）      → 关联前序事件
└── result（结果）     → 关联后续事件
```

| 角色 | 持有 | 物品 | 单向 | 张三-持有→玄天剑 |
| 角色 | 位于 | 地点 | 单向 | 张三-位于→青云峰 |
| 势力 | 附庸 | 势力 | 单向 | 青云宗-附庸→天剑门 |

### 2.2 自动知识提取（AI 驱动）

#### 2.2.1 对话中实时提取

**触发时机**：每次用户发送消息后（非流式，后台异步执行）

**提取流程**：

```
用户消息 + AI 回复
      ↓
LLM 提取任务（后台异步）
  - 识别提到的实体（新实体→创建，已有实体→更新属性）
  - 识别实体间的关系（新增/变化的关系）
  - 识别事件（战斗、交易、结盟等关键事件）
      ↓
生成提取结果（结构化 JSON）
      ↓
与现有图谱合并
  - 新实体：直接添加
  - 已有实体：合并属性（新值覆盖旧值，或标记冲突）
  - 新关系：添加边
  - 冲突检测：标注给用户确认
      ↓
图谱更新完成，触发 UI 刷新
```

**提取 Prompt 示例**：

```
你是一位知识图谱信息提取专家。从以下小说创作对话中提取所有实体、关系和事件。

返回 JSON 格式：
{
  "entities": [
    {"type": "Character", "name": "张三", "properties": {"gender": "男", "realm": "金丹期", "faction": "青云宗"}}
  ],
  "relations": [
    {"subject": "张三", "predicate": "师徒", "object": "李四", "direction": "bidirectional"}
  ],
  "events": [
    {"name": "青云峰论剑", "type": "battle", "subject": "张三", "object": "王五", "location": "青云峰", "result": "张三胜"}
  ]
}
```

#### 2.2.2 工作台条目批量提取

**入口**：工作台中的现有条目（世界观/角色/大纲）

**功能**：点击"提取到图谱"按钮，AI 解析已有条目的结构化内容，自动提取实体和关系

#### 2.2.3 待确认融合

将现有的"待确认"功能与图谱结合：

- 提取到新实体时，如果关键属性缺失，自动生成确认问题
- 确认结果直接更新图谱节点属性
- 例如：提取到"张三"但不知道境界 → 自动提问"张三的修炼境界是？"→ 用户回答"金丹期"→ 更新图谱

### 2.4 图谱驱动的 AI 创作

#### 2.4.1 图谱作为 AI 上下文

**核心改动**：在每次 AI 对话时，自动注入相关图谱知识作为 System Prompt 的一部分

```
System Prompt = 原始系统提示 + 图谱上下文

图谱上下文示例：
"""
【角色设定】
- 张三：男，金丹期，青云宗弟子，师父李四，持有玄天剑
- 李四：男，元婴期，青云宗长老，张三的师父
- 王五：女，筑基期，魔教圣女

【势力关系】
- 青云宗 vs 魔教：敌对
- 青云宗 vs 丹鼎宗：同盟

【关键事件】
- 青云峰论剑：张三击败王五（但王五暗中放水）
"""
```

**效果**：
- AI 不会把"金丹期"写成"筑基期"
- AI 知道张三和王五有交战经历，后续剧情合理衔接
- AI 不会让魔教和青云宗突然成为盟友（除非用户明确要求）

#### 2.4.2 一致性检测

**触发时机**：用户消息 + AI 回复生成后

**检测内容**：
- 境界矛盾（角色突然从金丹变成筑基，除非有明确降级事件）
- 关系矛盾（已死亡角色再次出现、敌对关系突然变友好）
- 地理矛盾（角色同时出现在两个相距遥远的地点）
- 时间线矛盾（事件顺序错误）

**处理方式**：检测到矛盾时，在工作台弹出警告卡片，用户可以选择：
- 接受（确实发生了变化，更新图谱）
- 拒绝（AI 写错了，重新生成）
- 修改（手动调整后更新）

#### 2.4.3 图谱辅助的创作建议

基于图谱的拓扑结构，AI 可以给出创作建议：

- **关系链挖掘**："张三的师父李四是青云宗长老，而王五是魔教圣女，他们的恋情属于禁忌，可以利用这个矛盾设计剧情转折"
- **事件推理**："张三击败王五后，魔教可能报复，建议设计一个'魔教围攻青云宗'的事件"
- **角色发展**："张三目前是金丹期，根据境界体系，下一步是元婴期突破，可以设计突破瓶颈的剧情"

### 2.5 项目迁移支持

#### 2.5.1 图谱导出

支持将整个知识图谱导出为结构化格式：

- **JSON**：完整图谱数据（节点 + 边 + 属性）
- **Markdown**：按实体类型分章节的文档
- **Mermaid**：可在 Markdown 中渲染的流程图/关系图

#### 2.5.2 图谱导入

新建项目时，可以导入已有项目的图谱，作为新创作的基础设定

## 三、技术方案

### 3.1 存储方案

**SQLite 图谱存储**（与现有架构一致）：

```sql
-- 图谱节点（实体）
CREATE TABLE graph_nodes (
    id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL,
    entity_type TEXT NOT NULL,     -- Character/Location/Faction/Item/Event
    name TEXT NOT NULL,
    properties TEXT DEFAULT '{}',  -- JSON 存储动态属性
    source_conversation_id TEXT,   -- 来源对话
    source_message_idx INTEGER,    -- 来源消息索引
    created_at TEXT,
    updated_at TEXT,
    FOREIGN KEY (project_id) REFERENCES projects(id)
);

-- 图谱边（关系）
CREATE TABLE graph_edges (
    id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL,
    subject_id TEXT NOT NULL,      -- 主体节点 id
    predicate TEXT NOT NULL,       -- 关系谓词
    object_id TEXT NOT NULL,       -- 客体节点 id
    direction TEXT DEFAULT 'directed',  -- directed / bidirectional
    properties TEXT DEFAULT '{}',  -- JSON 存储关系属性
    confidence REAL DEFAULT 1.0,   -- 置信度
    created_at TEXT,
    FOREIGN KEY (project_id) REFERENCES projects(id),
    FOREIGN KEY (subject_id) REFERENCES graph_nodes(id),
    FOREIGN KEY (object_id) REFERENCES graph_nodes(id)
);

-- 概念树
CREATE TABLE concept_tree (
    id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL,
    meta_concept TEXT NOT NULL,    -- 元概念：Realm/Rank/FactionLevel
    concept_name TEXT NOT NULL,
    parent_id TEXT,                -- 上级概念
    level INTEGER DEFAULT 0,
    FOREIGN KEY (project_id) REFERENCES projects(id),
    FOREIGN KEY (parent_id) REFERENCES concept_tree(id)
);

-- 矛盾记录
CREATE TABLE contradictions (
    id TEXT PRIMARY KEY,
    project_id TEXT NOT NULL,
    node_id TEXT,
    contradiction_type TEXT,       -- realm_conflict / relation_conflict / geo_conflict / timeline_conflict
    description TEXT,
    status TEXT DEFAULT 'pending', -- pending / accepted / rejected / resolved
    created_at TEXT
);
```

### 3.2 API 设计

```
图谱 CRUD
GET    /api/projects/{id}/graph/nodes          -- 获取所有节点（支持类型筛选）
POST   /api/projects/{id}/graph/nodes          -- 创建节点
PUT    /api/projects/{id}/graph/nodes/{nid}    -- 更新节点
DELETE /api/projects/{id}/graph/nodes/{nid}    -- 删除节点
GET    /api/projects/{id}/graph/edges          -- 获取所有边
POST   /api/projects/{id}/graph/edges          -- 创建边
DELETE /api/projects/{id}/graph/edges/{eid}    -- 删除边

AI 提取
POST   /api/projects/{id}/graph/extract        -- 从对话历史中提取实体和关系
POST   /api/projects/{id}/graph/extract-item/{item_id}  -- 从工作台条目提取

概念树
GET    /api/projects/{id}/concepts             -- 获取概念树
POST   /api/projects/{id}/concepts             -- 添加概念
PUT    /api/projects/{id}/concepts/{cid}       -- 更新概念
DELETE /api/projects/{id}/concepts/{cid}       -- 删除概念

一致性
GET    /api/projects/{id}/contradictions       -- 获取矛盾列表
PUT    /api/projects/{id}/contradictions/{cid} -- 处理矛盾

导出
GET    /api/projects/{id}/graph/export?format=json|markdown|mermaid

导入
POST   /api/projects/{id}/graph/import
```

### 3.3 前端技术选型

| 组件 | 方案 | 理由 |
|------|------|------|
| 图谱可视化 | **vis-network** | 轻量（~200KB），自带力导向布局、交互、事件系统，无需 React/Vue |
| 节点详情 | 现有 Modal 组件 | 复用已有弹窗系统 |
| 标签页切换 | 纯 CSS + JS | 与现有工作台标签页一致 |

### 3.4 提取策略

```
策略 1：实时提取（推荐初始方案）
- 每轮对话后异步调用提取接口
- 不阻塞用户操作
- 增量合并到图谱

策略 2：批量提取
- 用户手动触发"从对话提取"
- 一次性分析所有历史消息
- 适合项目初期快速构建图谱

策略 3：条目提取
- 从工作台已有的世界观/角色/大纲条目提取
- 已有结构化内容，提取精度更高
```

## 四、实施阶段

### Phase 1：基础图谱（MVP）
- 数据库表创建
- 节点和边的 CRUD API
- vis-network 基础可视化（力导向图）

- 概念语义推理（同境界角色、同地区势力）

## 五、与现有功能的融合点

| 现有功能 | 图谱融合方式 |
|---------|-------------|
| **工作台条目** | 条目 ↔ 图谱节点双向同步；从条目提取到图谱，从图谱生成条目 |
| **待确认（❓）** | 提取到不完整实体时自动触发确认；确认结果直接更新图谱 |
| **灵感（✨）** | 灵感总结时自动关联图谱中的已有实体，提示"这个灵感与张三有关" |
| **AI 添加（+）** | AI 生成工作台条目时，同步提取实体到图谱 |
| **助理系统** | 不同助理关注不同图谱子集（世界观助理关注地点/势力，角色助理关注人物关系） |
| **编年史** | 事件节点按时间排序就是编年史；图谱的事件因果链就是剧情线 |

---

## 六、风险与注意事项

1. **性能**：大图谱（1000+ 节点）的前端渲染性能 → vis-network 支持虚拟渲染，但需要控制显示密度
2. **提取准确率**：LLM 提取的实体和关系可能有误 → 必须有用户确认环节，不能完全自动
3. **存储增长**：长篇小说可能有大量节点 → 按项目隔离，支持归档
4. **学习成本**：用户可能不熟悉图谱概念 → 用直观的 UI 包装，用户无需理解底层图论
5. **AI 成本**：每次对话后额外调用一次 LLM 提取 → 可以设置提取频率（每 3 轮提取一次），或用户手动触发
