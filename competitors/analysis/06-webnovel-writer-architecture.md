# webnovel-writer 技术架构深度分析

## 一、项目概览

| 属性 | 内容 |
|------|------|
| **项目名称** | webnovel-writer |
| **GitHub** | https://github.com/lingfengQAQ/webnovel-writer |
| **版本** | v5.5.4 |
| **许可证** | GPL v3 |
| **定位** | 基于 Claude Code 插件系统的长篇网文 AI 辅助写作系统 |

面向中文网络文学创作者的 AI 写作增强插件，构建在 Anthropic Claude Code 之上，通过 Skills、Agents、References 三层架构，提供从大纲规划、章节撰写到质量审查的全流程写作支持。支持 37+ 种网文题材，允许最多两种题材复合叠加（7:3 比例）。

---

## 二、技术栈

| 组件 | 技术 | 说明 |
|------|------|------|
| **LLM** | Claude (Claude Code 内置) | 通过 Claude Code 插件系统调用 |
| **嵌入** | Qwen3-Embedding-8B | 支持本地部署或 API 调用 |
| **重排序** | jina-reranker-v3 | RAG 结果精排 |
| **向量存储** | SQLite BLOB | struct.pack 存储浮点数组，无需独立向量数据库 |
| **BM25** | 自研实现 | 中文字符 + 英文单词分词 |
| **图数据库** | SQLite 关系表 | expand_hops=1 轻量级图 RAG |
| **Web 框架** | FastAPI + Uvicorn | 仅用于只读仪表盘，端口 8765 |
| **状态管理** | JSON + SQLite | state.json + index.db |
| **配置** | .env + dataclass | 100+ 可调参数 |

**设计亮点**：整个技术栈无任何重量级外部依赖，向量存储、BM25、图 RAG 全部基于 SQLite 实现。

## 四、Agent / 工作流设计

### 4.1 双 Agent 架构（核心设计）

**读写分离模式**，避免单个 Agent 在同一上下文中既要理解大量历史数据又要生成新内容的注意力分散：

| Agent | 职责 | 输入 | 输出 |
|-------|------|------|------|
| **Context Agent** (只读) | 读取项目状态、组装写作上下文、时间线构建、追读力分析、红线校验 | state.json, index.db, summaries/ | Context Pack (JSON) |
| **Data Agent** (只写) | AI 实体提取、实体消歧、写入 DB、场景分块、向量索引、摘要生成、债务利息计算 | Context Pack + 新章节文本 | 更新后的 index.db, state.json, summaries/, vectors.db |

**Context Agent 6 步管道**：Step 0 快照 → Step 0.5 契约 → Step 0.6 时间线 → Step 1 大纲+状态 → Step 2 追读力+债务 → Step 3 实体+伏笔 → Step 4 摘要+推理 → Step 5 组装 → Step 6 红线校验

**Data Agent 10 步管道**：Step A 加载 → Step B AI 实体提取 → Step C 消歧 → Step D 写入 DB → Step E 摘要 → Step F 场景分块 → Step G 向量索引 → Step H 风格样本 → Step I 债务利息 → Step J 计时报告

**三级置信度消歧**：>0.8 自动采纳 / 0.5-0.8 采纳+警告 / <0.5 挂起等待人类确认

### 4.2 六维并行审查系统

| 检查器 | 类型 | 检查内容 |
|--------|------|----------|
| **一致性 Checker** | 核心 | 战力一致、地点一致、角色一致 |
| **连续性 Checker** | 核心 | 时间线、因果链、伏笔回收 |
| **OOC Checker** | 核心 | 轻微偏离、中度失真、严重崩坏（区分角色成长 vs OOC） |
| **追读力 Checker** | 条件(默认) | 钩子设计、信息差构建、微观回报 |
| **爽点 Checker** | 条件(5章) | 8种爽点模式、30/40/30结构、滚动窗口密度 |
| **节奏 Checker** | 条件(10章) | Quest/Fire/Constellation 三线平衡 |

问题分级：Critical(阻断) → High(必须修) → Medium/Low(建议)

## 五、核心功能模块

### 5.1 防幻觉三定律

| 定律 | 内容 |
|------|------|
| **大纲即法律** | 不得偏离已确认大纲，未规划情节不得自行发明 |
| **设定即物理** | 世界观设定如同物理定律不可违反，疑似冲突暂停请求人类裁决 |
| **发明需识别** | 创造新设定/角色必须明确标注，低置信度发明挂起等待确认 |

约束层分级：Hard（必须执行）/ Soft（建议执行）/ Style（可选执行）

### 5.2 Strand Weave 三线编织系统

| 线 | 比例 | 职责 | 红线约束 |
|----|------|------|----------|
| **Quest (主线)** | 60% | 推进核心剧情 | 最大连续 5 章 |
| **Fire (爽线)** | 20% | 高潮/打脸/突破 | 最大间隔 10 章 |
| **Constellation (伏线)** | 20% | 伏笔/暗示/世界观展开 | 最大间隔 15 章 |

### 5.3 追读力评分系统

- **5 种钩子**: 危机钩、悬念钩、渴望钩、情绪钩、选择钩
- **8 种爽点**: 装逼打脸、扮猪吃虎、越级反杀、打脸权威、反派翻车、甜蜜超预期、迪化误解、身份掉马
- **7 种微观回报**: 小悬念揭示、小目标达成、新能力展示、关系进展、信息差揭晓、反派受挫、世界观碎片
- **硬性不变量**: HARD-001 每章至少 1 个钩子 / HARD-002 每 3 章至少 1 个微观回报 / HARD-003 爽点密度不低于题材基线 / HARD-004 压扬比不超过阈值

### 5.4 6 步写作管道

```
Step 1 (Context) → Step 2A (Draft) → Step 2B (Style) → Step 3 (Review) → Step 4 (Polish) → Step 5 (Data Persist) → Step 6 (Git)
```

硬约束：不可合并/跳过/重命名步骤，不可自审。Step 5 故障隔离：G/H 子步骤失败不回滚 Step 1-4。

### 5.5 RAG 管道

```
查询 → QueryRouter(auto) → Vector Search(Top-30) + BM25(Top-20) + Graph-Hybrid(种子→1-hop扩展)
→ RRF 融合(k=60) → Rerank(jina-v3, Top-10)
```

大规模优化：>500 向量时 BM25 预过滤 + 最近 chunk 优先。降级模式：嵌入认证失败回退纯 BM25。

## 六、数据流与存储

### 存储架构

| 存储 | 内容 |
|------|------|
| **state.json** | 当前章节号、卷、strand_tracker、debt_tracker、genre、项目元信息 |
| **index.db (SQLite)** | entities(实体表)、relationships(关系表)、state_changes(状态变更)、review_metrics(审查指标)、foreshadowing(伏笔，三级：core/sub/decor) |
| **vectors.db (SQLite)** | chunks(文本块+embedding BLOB)、bm25_index(倒排索引) |
| **文件系统** | summaries/(章节摘要)、context_snapshots/(上下文快照)、大纲/、设定集/、call_trace.jsonl |

### 完整数据流

```
用户CLI命令 → Skill入口 → Context Agent(读state.json+index.db+summaries+大纲+设定 → Context Pack)
→ Claude写作(初稿→风格化→6维审查→修改抛光) → Data Agent(实体提取→消歧→DB写入→场景分块→向量索引→摘要→债务计算)
→ Git提交 → 完成反馈
```

## 七、亮点与创新点

1. **Prompt-as-Code 架构**: Agent 全部通过 Markdown 文件定义，编辑 .md 即可调整 Agent 行为
2. **读写分离双 Agent**: 解决 LLM 长上下文中读写干扰问题
3. **故障隔离设计**: 数据持久化失败不回滚创作成果
4. **追读力分类法**: 5 钩子 + 8 爽点 + 7 微观回报 + 压扬比的形式化评估体系
5. **Strand Weave 三线编织**: 将叙事学理论转化为可执行约束
6. **债务追踪系统**: 弹性约束满足，违反记录债务，后续超额完成偿还
7. **纯 SQLite 全栈**: 向量存储、BM25、图 RAG 零外部依赖
8. **三级置信度消歧**: 人机协同的渐进式信任模型
9. **Context Contract v2**: 优先级排序 + 新鲜度/频率/钩子加分 + 动态 Token 预算 + 分阶段预算
10. **中文思维写作约束**: 禁止"先英后中"，要求直接用中文思维创作

## 八、对 Fiction Academy 的启示

### 可直接借鉴

| 设计 | 启示 |
|------|------|
| Prompt-as-Code | Agent 行为通过 Markdown 定义，修改成本低 |
| 读写分离 Agent | 专职"读组装"和"写提取"的 Agent 分离，降低上下文污染 |
| 故障隔离管道 | 每个步骤独立故障边界，数据持久化失败不回滚创作成果 |
| 债务追踪机制 | 弹性约束系统，违反记录债务，后续超额偿还 |
| 形式化创作理论 | 追读力、爽点、节奏控制转化为可执行规则 |
| 约束层分级 | Hard / Soft / Style 三级，可扩展更多层级 |
| 题材画像系统 | 题材作为参数化配置（压扬比、爽点基线、节奏偏好） |

### 差异化超越方向

| 维度 | webnovel-writer | Fiction Academy 方向 |
|------|-----------------|---------------------|
| Agent 协作 | 单 Agent 顺序执行 | 真正的多 Agent 并行协作 |
| 实时交互 | CLI 命令批处理 | 实时协作编辑、流式审查 |
| 扩展性 | Checker/Skill 固定 | 插件化审查器/Agent 注册机制 |
| RAG 能力 | SQLite + 1-hop Graph | 更强知识图谱 + 多跳推理 |
| 跨作品 | 单项目隔离 | 跨作品知识迁移 |

**核心启示**: AI 创作系统的核心竞争力不是模型能力，而是**领域知识的工程化深度**。谁能把创作理论转化为更精确、更可执行、更可量化的规则系统，谁就能建立真正的护城河。
